**Part C: Update theme.ts store**

Replace the entire content of `apps/web/src/lib/stores/theme.ts` with:

```typescript
import { writable } from 'svelte/store';
import { browser } from '$app/environment';

export type Theme = 'lila' | 'hell' | 'dunkel';

export const themes: { id: Theme; label: string; color: string }[] = [
  { id: 'lila',   label: 'Lila',   color: '#80377B' },
  { id: 'hell',   label: 'Hell',   color: '#475569' },
  { id: 'dunkel', label: 'Dunkel', color: '#1E293B' },
];

const stored = browser ? (localStorage.getItem('theme') as string | null) : null;
const initial: Theme = (stored && themes.find(t => t.id === stored)) ? stored as Theme : 'lila';

export const theme = writable<Theme>(initial);

theme.subscribe(value => {
  if (!browser) return;
  localStorage.setItem('theme', value);
  document.documentElement.setAttribute('data-theme', value);
});
```

Key changes:
- Type narrowed to `'lila' | 'hell' | 'dunkel'`
- `themes` array has 3 entries with correct labels and preview colors
- Default fallback is `'lila'` (not 'pflaume')
- Old localStorage values ('pflaume', 'nacht', 'wald', 'schiefer', 'pro') won't match any `themes.find()`, so they gracefully fall back to 'lila'
- `stored` typed as `string | null` (not `Theme | null`) so old values don't cause type issues

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-03 | Tampering | localStorage theme value | accept | Theme value is validated against known themes array via `themes.find()`. Unknown values fall back to 'lila'. The `data-theme` attribute only selects CSS variable sets — no code execution. Worst case: an invalid value means no theme overrides apply and :root defaults are used. |
| T-08-04 | Spoofing | CSS injection via data-theme | accept | The `data-theme` attribute value is set programmatically from a validated store value, not from user input.
