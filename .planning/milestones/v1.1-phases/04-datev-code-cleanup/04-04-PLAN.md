# Plan 4: Add DATEV Export section to Systemeinstellungen admin UI

## Goal

Add a new "DATEV Export" section to `apps/web/src/routes/(app)/admin/system/+page.svelte`,
positioned BEFORE the Phorest-Integration section (per D-11), using the existing
`section-label` + `card card-body settings-card` pattern (per D-12). The section
contains four German-labelled number inputs (Normalstunden, Urlaub, Krank / AU,
Sonderurlaub) bound to the 4 TenantConfig fields added by Plan 2, loaded via
`GET /settings/work` and saved via `PUT /settings/work` (per D-13 corrected by
research). This plan completes DATEV-03 end-to-end.

## Threat Model (ASVS L1)

| Threat | ASVS | Mitigation | Implemented |
|--------|------|------------|-------------|
| Access control: non-admin sees or modifies DATEV Lohnarten | V4.1 | The entire `/admin/system` route is guarded by the existing layout-level auth (admin-only); the API endpoint `PUT /settings/work` already has `requireRole("ADMIN")` — backend is authoritative. Non-admin cannot reach this page. | already in place at layout + API level |
| Input validation on client: user enters negative / huge numbers | V5.1 | Use `<input type="number" min="1" max="9999" step="1">` AND rely on the Zod schema in settings.ts (min/max enforced by Plan 2) as the authoritative check. Client validation is UX-only, not a security boundary. | enforced on both sides |
| XSS via reflected config values | V5.3 | Svelte auto-escapes `{value}` interpolation; all labels are static German strings; no `{@html}` used | safe by default |
| CSRF on PUT /settings/work | V4.3 | Existing JWT-in-httpOnly-cookie + SvelteKit fetch pattern already in place (see `saveFederalState` at line 357); inherit the pattern | already in place |
| Info disclosure: storing stale config in localStorage | V8 | Values are only held in component `$state` for the duration of the page view; not persisted client-side | safe by default |

No new attack surface. The new section is a thin form over an already-authenticated API.

**Step 4 — Add a `saveDatev()` function** in the `<script>` block, modeled EXACTLY on `saveFederalState()` (lines 351–369). Do NOT spread `_gOtherFields` — the PUT /work handler accepts partial updates (all fields are `.optional()` in `tenantConfigSchema`), so sending only the 4 DATEV fields is correct and safer:

```typescript
async function saveDatev() {
  datevSaving = true;
  datevError = "";
  datevSaved = false;
  try {
    await api.put("/settings/work", {
      datevNormalstundenNr,
      datevUrlaubNr,
      datevKrankNr,
      datevSonderurlaubNr,
    });
    datevSaved = true;
    setTimeout(() => (datevSaved = false), 3000);
  } catch (e: unknown) {
    datevError = e instanceof Error ? e.message : "Fehler";
  } finally {
    datevSaving = false;
  }
}
```
