# Plan 2: Add DATEV Lohnarten fields to TenantConfig schema and settings API

## Goal

Add the four configurable DATEV Lohnartennummern to the `TenantConfig` Prisma
model (`datevNormalstundenNr`, `datevUrlaubNr`, `datevKrankNr`,
`datevSonderurlaubNr`), extend the existing `tenantConfigSchema` in
`apps/api/src/routes/settings.ts` so that `PUT /settings/work` accepts them, and
ensure `GET /settings/work` returns them (with defaults `100/300/200/302`).
This is the backend half of DATEV-03 — it lays the storage + API foundation so
Plan 3 can read the values and Plan 4 can bind the UI. No `reports.ts` changes
happen in this plan; that is Plan 3's scope.

## Threat Model (ASVS L1)

| Threat | ASVS | Mitigation | Implemented |
|--------|------|------------|-------------|
| Tampering: non-admin writes arbitrary Lohnartennummern, corrupting payroll export | V4.1 Access Control | `PUT /settings/work` is already gated by `requireRole("ADMIN")` (settings.ts line 155) | already in place — no change |
| Input validation: negative or absurdly large integers break DATEV import | V5.1 Input Validation | `z.number().int().min(1).max(9999).optional()` on each of the 4 fields in `tenantConfigSchema` | enforced by new schema rules |
| Tenant isolation: cross-tenant write via forged tenantId | V4.2 | Handler derives `tenantId` from `getTenantId(app, req.user.sub)` — never accepts from body | already in place |
| Audit trail: silent modification of payroll-critical config | V7.1 Logging | `app.audit({ action: "UPDATE", entity: "TenantConfig", newValue: body, ... })` already runs at line 234 and will include the new fields via `...body` spread | already in place |
| Data integrity: partial write leaves inconsistent state (3 of 4 fields saved) | V5.3 | `prisma.tenantConfig.upsert` is atomic; all 4 fields land in a single transaction | already enforced by Prisma |

No new attack surface. The 4 new fields are non-sensitive small integers. The `requireRole("ADMIN")` gate on the existing route is sufficient for V4.
